# 2.5 Repository Design

**Input:** 0.12 persistence discovery (11 idb-keyval files) + 2.2 Boundary 4
**Protocol:** Evidence-First

## Design Decision

**No port needed for persistence.** A service wrapper is sufficient.

**Evidence:**
- 0.13: idb-keyval is the only persistence technology
- No evidence of planned replacement (SQLite, remote API, etc.)
- Adding a port adds 2 files (port + adapter) for zero benefit

## Persistence Service Design

Create a single `persistence/` module that wraps idb-keyval:

```
services/persistence/
├── index.ts          ← re-export hub
├── campaignStore.ts  ← campaign CRUD (moved from store/)
├── loreStore.ts      ← lore chunk CRUD (moved from store/campaignStore.ts)
├── npcStore.ts       ← NPC ledger CRUD (moved from store/campaignStore.ts)
├── settingsStore.ts  ← settings persistence (moved from store/slices/settingsSlice.ts)
├── imageStore.ts     ← image CRUD (already in services/storage/)
├── embeddingStore.ts ← embedding CRUD (already in services/storage/)
└── archiveStore.ts   ← archive CRUD (already in services/storage/)
```

## What Moves Where

| Current Location | Responsibility | New Location |
|-----------------|---------------|--------------|
| store/campaignStore.ts | Campaign CRUD + Lore CRUD + NPC CRUD + Pressure CRUD + Archive load + Divergence save | services/persistence/ |
| store/slices/settingsSlice.ts (idb parts) | Settings save/load | services/persistence/settingsStore.ts |
| services/storage/imageStorage.ts | Image CRUD | services/persistence/imageStore.ts (rename) |
| services/storage/embeddingStorage.ts | Embedding CRUD | services/persistence/embeddingStore.ts (rename) |
| services/storage/archiveStorage.ts | Archive CRUD | services/persistence/archiveStore.ts (rename) |
| services/storage/backupStorage.ts | Backup CRUD | services/persistence/backupStore.ts (rename) |
| services/campaignBundle.ts | Bundle export/import | services/persistence/bundleStore.ts (rename) |

## What Stays in Store

After extraction, store/campaignStore.ts becomes EMPTY — all its
functions move to services/persistence/. Store slices keep only
reactive state (get/set/subscribe), no persistence.
