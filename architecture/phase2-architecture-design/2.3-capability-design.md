# 2.3 Capability Design

**Input:** 0.6 capability discovery (930 exports, 56 clusters) + 2.2 boundaries
**Protocol:** Evidence-First

## Design Principle

Capabilities are designed from what the code ACTUALLY does (0.6),
not from what patterns suggest. Each capability maps to a real
cluster of exports with a shared domain concern.

## Capability Design (from evidence)

### Cap-1: Messaging
- **Exports:** addMessage, updateLastAssistant, updateLastMessage, setCondensed, setStreaming, replaceMessageText, deleteMessage, etc.
- **Owner:** state (chatSlice)
- **Consumers:** turn orchestrator, ChatArea, pendingCommit
- **Boundary:** Boundary 1 (Domain↔State)
- **Needs port?** ✅ Yes — 9 service files import store for messaging

### Cap-2: NPC Management
- **Exports:** addNPC, updateNPC, removeNPC, setOnStageNpcIds, addNpcSuggestions, mergeOrRenameNpc, applyPressurePatch
- **Owner:** state (npcSlice)
- **Consumers:** npcGeneration, turnPostProcess, image/portrait, pendingCommit
- **Boundary:** Boundary 1
- **Needs port?** ✅ Yes — 4 service files import store for NPC ops

### Cap-3: Archive & Divergence
- **Exports:** setChapters, setArchiveIndex, setSemanticFacts, clearPinnedChapters, setDivergenceRegister, updateMessageDivergence, toggleDivergenceChapter
- **Owner:** state (archiveSlice + chatSlice divergence section)
- **Consumers:** turnPostProcess, pendingCommit, MemoryTab
- **Boundary:** Boundary 1
- **Needs port?** ✅ Yes — turnPostProcess and pendingCommit import store

### Cap-4: Campaign Context
- **Exports:** updateContext, incrementBookkeepingTurnCounter, resetBookkeepingTurnCounter, context (read), activeCampaignId (read)
- **Owner:** state (campaignSlice)
- **Consumers:** turnPostProcess, pendingCommit, backfillRunner
- **Boundary:** Boundary 1
- **Needs port?** ✅ Yes — 3 service files import store for context

### Cap-5: Settings (read-only for services)
- **Exports:** getSettings, getActivePreset, getActiveImageEndpoint, getActiveTier
- **Owner:** state (settingsSlice)
- **Consumers:** image services, embedder, llmService, turnOrchestrator
- **Boundary:** Boundary 1
- **Needs port?** ✅ Yes — services read settings from store

### Cap-6: Notification
- **Exports:** success, error, warning, info
- **Owner:** ui (Toast component)
- **Consumers:** all services + all store slices
- **Boundary:** Boundary 2 + Boundary 3
- **Needs port?** ✅ Yes — 5 services + 3 store slices import Toast

### Cap-7: Persistence
- **Exports:** saveCampaignState, loadCampaignState, saveLoreChunks, getLoreChunks, saveNPCLedger, getNPCLedger, savePressure, saveDivergenceRegister
- **Owner:** state (campaignStore) + domain (storage/*)
- **Consumers:** store slices, services, components
- **Boundary:** Boundary 4
- **Needs port?** ⚠️ Maybe — see 2.2 OQ2. Evidence suggests service is sufficient (idb-keyval is final).

### Cap-8: UI State (transient)
- **Exports:** setPipelinePhase, setStreamingStats, setLastPayloadTrace
- **Owner:** state (uiSlice)
- **Consumers:** turnOrchestrator, pendingCommit
- **Boundary:** Boundary 1
- **Needs port?** ⚠️ Questionable — these are transient UI hints. Could be event-based or callback-based instead.

## Capabilities that DON'T need ports

### Cap-9: NPC Generation (internal)
- Pure domain logic, no state access needed (receives data via parameters)
- No boundary violation — stays in domain layer

### Cap-10: Turn Orchestration (internal)
- Pipeline with 13 stages — already well-structured
- Needs ports ONLY for communicating with state (covered by Cap-1 through Cap-5)

### Cap-11: Embedding (internal)
- Self-contained worker + storage
- Needs Cap-5 (Settings) for model config — covered

### Cap-12: LLM Communication (internal)
- Pure network calls
- No state access — no port needed

### Cap-13 through Cap-18: All internal
- Engine, Lore, Arc, Image, Infrastructure, TTS
- These are domain-internal — no boundary violations

## Summary

| Capability | Needs port? | Evidence |
|------------|------------|----------|
| Messaging | ✅ Yes | 9 service→store imports |
| NPC Management | ✅ Yes | 4 service→store imports |
| Archive & Divergence | ✅ Yes | 3 service→store imports |
| Campaign Context | ✅ Yes | 3 service→store imports |
| Settings (read) | ✅ Yes | 4 service→store imports |
| Notification | ✅ Yes | 8 imports (5 services + 3 store) |
| Persistence | ⚠️ Maybe | Service sufficient (idb-keyval is final) |
| UI State | ⚠️ Questionable | Event/callback may be better |

**6 capabilities definitely need ports. 2 are questionable.**

## Open Questions

1. Should Persistence be a port or a service? → 2.5 answers
2. Should UI State be port, event, or callback? → 2.7 answers
3. Should NPC Management be one port or split (NPC + Pressure)? → Evidence shows they share the same owner (npcSlice) and consumers. One port.
