# 2.2 Boundary Design

**Input:** 2.1 validated targets + 0.5 boundary discovery + 0.15 violations
**Protocol:** Evidence-First вҖ” boundaries designed from violations, not from patterns

## Design Principle

A boundary is needed ONLY where a violation exists. We don't create
boundaries for theoretical purity вҖ” we create them to fix real problems
found in the code.

## Boundary 1: Domain вҶ” State Boundary

**Problem:** 16 domainвҶ’state + 28 stateвҶ’domain = 44 violations
**Evidence:** 0.15/RAW_DATA.json (44 of 53 total violations)

**Design:**
- Domain must NOT import store directly
- State must NOT import services directly
- Communication happens through contracts (ports)

**Which direction first?**
- domainвҶ’state (16 violations): Easier вҖ” services already know what they need
- stateвҶ’domain (28 violations): Harder вҖ” store contains business logic that must move

**Decision:** Design ports for domainвҶ’state first. StateвҶ’domain requires
logic extraction (Phase 3), not just boundary design.

## Boundary 2: Domain вҶ” UI Boundary

**Problem:** 6 domainвҶ’ui violations
**Evidence:** 0.15/RAW_DATA.json

**Design:**
- Services must NOT import components
- The 6 imports are: toast notifications (5) + useMessageEditor (1)
- Solution: notification port (already proven in candidate architecture)
  + extract useMessageEditor's findSceneIdForMessage to a service

## Boundary 3: State вҶ” UI Boundary

**Problem:** 3 stateвҶ’ui violations
**Evidence:** 0.15/RAW_DATA.json

**Design:**
- Store must NOT import components
- The 3 imports are toast from components/Toast.tsx
- Solution: notification port (same as Boundary 2)

## Boundary 4: Persistence Gateway Boundary

**Problem:** 11 files access idb-keyval directly, no gateway
**Evidence:** 0.12/RAW_DATA.json

**Design:**
- Create a persistence layer that wraps idb-keyval
- All other modules import from this layer
- campaignStore delegates to persistence, doesn't own it

## Boundary 5: God File Internal Boundaries

**Problem:** 9 God Files with multiple responsibilities
**Evidence:** 2.1 validation

**Design:**
- npcGeneration вҶ’ split by responsibility (gen, validate, PC, agency, drives)
- turnPostProcess вҶ’ split by stage (archive, witness, NPC, seal, telemetry)
- chatSlice вҶ’ split by data domain (messages, condenser, divergence, excerpts)
- campaignSlice вҶ’ extract domain logic, keep only state
- God Components вҶ’ split by concern (render, state, actions)

## Open Questions

1. Should Boundary 1 (DomainвҶ”State) use ports, events, or callbacks?
   - Ports: explicit contracts, testable, but adds 10+ files
   - Events: looser coupling, but harder to trace
   - Callbacks: already used (TurnCallbacks), but hidden
   вҶ’ Needs 2.4 Port Design to answer

2. Should Boundary 4 (Persistence) be a port or a service?
   - Port: if persistence might be replaced (SQLite, remote)
   - Service: if idb-keyval is the final choice
   вҶ’ Evidence: 0.13 shows idb-keyval is the only persistence. Service is sufficient.

3. Should Boundary 5 (God Files) be designed here or in Phase 3?
   - Here: gives clear split plan
   - Phase 3: more flexible
   вҶ’ Design split plan here, execute in Phase 3
