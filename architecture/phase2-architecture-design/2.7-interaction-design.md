# 2.7 Interaction Design

**Input:** 0.10 (624 callback patterns) + 0.11 (174 store operations) + 2.4 ports
**Protocol:** Evidence-First

## Interaction Patterns (from evidence)

### Pattern 1: Port-based (designed in 2.4)
- Service calls port → adapter → store
- Used for: messaging, NPC, archive, context, settings, notification
- 31 violations fixed this way

### Pattern 2: Callback-based (already exists — TurnCallbacks)
- UI creates callback object → passes to service
- Used for: TurnCallbacks (21 callbacks)
- No violation — callbacks are parameters, not imports

**Decision:** Keep TurnCallbacks as-is. It's not a violation.
The callback provider (ChatArea) will delegate to ports instead of
direct store access, but the callback pattern itself stays.

### Pattern 3: Direct store access (UI → store)
- Components use useAppStore hook
- 18 component→store imports in 0.4
- No violation — UI reading state is expected

### Pattern 4: Dynamic import (lazy loading)
- 73 dynamic imports in 0.9
- Used for: code splitting, circular dependency avoidance
- No violation — technical optimization

## Interaction Flow (designed)

```
User taps "Send"
  → ChatArea creates TurnCallbacks (delegating to ports)
  → turnOrchestrator.runTurn(callbacks)
    → stages call llmCall, embedText, etc. (domain-internal)
    → stages call callbacks.addMessage() → port → adapter → store
    → store notifies subscribers → UI re-renders

Turn completes
  → pendingCommit.commitPendingTurn()
    → calls ports (messaging, npc, archive, context)
    → ports → adapters → store
    → store triggers debounced save (via persistence service)
```

## What Changes from Current

| Current | Designed |
|---------|----------|
| Services import store directly | Services import ports |
| Store slices import services | Store slices only have state (logic extracted) |
| ChatArea builds callbacks from store | ChatArea builds callbacks from ports |
| 11 files access idb-keyval | 1 persistence service accesses idb-keyval |
| 53 violations | 0 violations (after full Phase 3) |

## Open Questions

1. Should the callback pattern (TurnCallbacks) be replaced with ports?
   → No. Callbacks are parameters, not imports. No violation.
   → But the callback PROVIDER should use ports instead of direct store access.

2. Should there be an event bus for UI state updates?
   → No evidence supports this. Current callback pattern works.
   → Only add if Phase 3 shows the pattern is insufficient.
