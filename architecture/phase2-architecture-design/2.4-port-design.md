# 2.4 Port Design

**Input:** 2.3 (6 confirmed capabilities need ports) + 0.15 violations
**Protocol:** Evidence-First — each port method maps to a real violation

## Design Principle

A port method exists ONLY if a real import violation proves it's needed.
No speculative methods. No "might be useful" methods.

## Ports NOT Designed (with evidence)

### PersistencePort — NOT NEEDED
**Evidence:** 0.13 shows idb-keyval is the only persistence layer.
No evidence of planned replacement. A service wrapper is sufficient.
**Decision:** Use a persistence service, not a port.

### UIStatePort — NOT NEEDED
**Evidence:** Only 3 methods (setPipelinePhase, setStreamingStats, setLastPayloadTrace).
These are transient UI hints, not domain state.
**Decision:** Use callback parameters (like TurnCallbacks already does).
No port needed — the callback pattern is already in use.

## Open Questions

1. Should the remaining 22 state→domain violations be fixed with ports (reverse direction) or logic extraction?
   → Logic extraction. Store should not call services. Services should be called by UI or orchestrator.
   → This is Phase 3 work, not Phase 2 design.

2. Should PersistencePort be added if persistence might change?
   → No. YAGNI. Add when evidence shows a need.
