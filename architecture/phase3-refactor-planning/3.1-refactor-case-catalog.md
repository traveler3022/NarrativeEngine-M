# 3.1 Refactor Case Catalog

**Input:** Phase 0 evidence (0.15 violations, 0.16 God Files, 0.12 persistence, 0.8 state ownership) + Phase 2 design (2.1–2.9)
**Protocol:** Evidence-First — every RF case traces to specific Phase 0 evidence and a Phase 2 design decision

## Catalog Convention

Each refactor case (RF-XXX) is a **self-contained problem statement** with:
- Evidence pointer (which Phase 0/1/2 artifact proves the problem exists)
- Root cause (why the problem exists, not just what)
- Candidate solutions considered (not just the chosen one)
- Best solution with justification (why this option, not others)
- Risk (what can go wrong during implementation)
- Dependencies (other RF cases that must be completed first)
- Files affected (concrete file list)
- Estimated wave (which implementation wave executes this)

Cases are grouped by **problem class**, not by solution type. A boundary
violation and a God File are different problem classes even if both are
fixed by extraction.

## Problem Classes

| Class | Description | Cases | Solution Family |
|-------|-------------|-------|-----------------|
| A. Boundary violation (domain→state) | Services import store directly | RF-001..RF-005 | Port |
| B. Boundary violation (→ui) | Services or slices import components | RF-006, RF-007 | Port (Notification) |
| C. Boundary violation (state→domain) | Store slices contain business logic | RF-008..RF-010 | Logic extraction |
| D. Persistence scatter | idb-keyval accessed from 11 places | RF-011 | Service consolidation |
| E. God File (domain) | Domain file with multiple responsibilities | RF-012, RF-013 | Internal split |
| F. God File (state) | Slice file with multiple data domains | RF-014 | Internal split |
| G. God Component | UI file with multiple concerns | RF-015..RF-019 | Internal split |

**19 RF cases total, covering all 53 violations + 9 God Files + 11 persistence access points.**

## Class A: Domain→State Boundary Violations

### RF-001: Messaging services import store directly

| Field | Value |
|-------|-------|
| ID | RF-001 |
| Problem | Domain services import `useAppStore` to append messages, update streaming, condense history |
| Evidence | 0.15/RAW_DATA.json — 9 violations across 6 service files |
| Impact | High — services can't be tested without store; coupling blocks logic extraction |
| Root Cause | Services were written assuming store is always available; no contract was defined between domain and state |
| Candidate Solutions | (a) Port (b) Pass store via parameter (c) Event bus (d) Callbacks |
| Best Solution | **MessagingPort** (10 methods) |
| Why | Port gives explicit contract, testable in isolation, matches Phase 2.4 evidence. (b) pollutes every service signature; (c) hides flow; (d) already used for TurnCallbacks but not for state mutation |
| Risk | Low — adapter is thin delegate, no logic moves |
| Dependencies | None (W0 — infrastructure) |
| Files | services/turn/pendingCommit.ts, services/turn/turnPostProcess.ts, services/turn/turnTypes.ts, services/engine/engineRolls.ts, services/image/index.ts, services/image/portrait.ts |
| Estimated Wave | W0 (port+adapter), W1 (service migration) |

## Class C: State→Domain Boundary Violations (Logic Extraction)

### RF-008: campaignSlice contains business logic (9 violations)

| Field | Value |
|-------|-------|
| ID | RF-008 |
| Problem | campaignSlice imports 9 services (embedding, storage, campaign-state, turn, infrastructure, apiClient) to do domain work |
| Evidence | 0.15/RAW_DATA.json — 9 state→domain violations in campaignSlice.ts |
| Impact | High — slice is doing work that should live in services or orchestrator |
| Root Cause | Slice was extended incrementally; each new feature added service imports instead of moving logic out |
| Candidate Solutions | (a) Extract logic to services, slice keeps only state (b) Move logic to orchestrator (c) Keep in slice but pass services via parameters |
| Best Solution | **Logic extraction to services** — slice keeps only `set()`/`get()` |
| Why | Slice should be reactive state only; orchestrator/services own domain logic. (b) is partial — some logic doesn't belong in turn orchestrator. (c) doesn't fix coupling. |
| Risk | Medium — logic moves, must preserve call sites and timing |
| Dependencies | RF-001..RF-005 (ports must exist so extracted logic can call back) |
| Files | store/slices/campaignSlice.ts (1 file, ~150 LOC moved) |
| Estimated Wave | W4 |

### RF-009: campaignStore is a God Module (8 violations + 7 responsibilities)

| Dependencies | RF-008 (campaignSlice must be pure state first), RF-001..RF-005 (ports for any domain calls) |

## Class D: Persistence Scatter

### RF-011: idb-keyval accessed from 11 files

| Field | Value |
|-------|-------|
| ID | RF-011 |
| Problem | 11 files import idb-keyval directly; no gateway; persistence technology is hardcoded everywhere |
| Evidence | 0.12/RAW_DATA.json — 11 access points |
| Impact | Medium — not a runtime bug, but blocks future migration (e.g., to SQLite) |
| Root Cause | No persistence abstraction was ever created; each new feature imported idb-keyval directly |
| Candidate Solutions | (a) PersistencePort + adapter (b) Persistence service (no port) (c) Keep as is |
| Best Solution | **Persistence service** (no port — per Phase 2.5) |
| Why | Phase 0.13 shows idb-keyval is the only persistence technology with no replacement planned. Port would be speculative (YAGNI). Service gives single access point without port overhead. |
| Risk | Low — service is a wrapper, no logic changes |
| Dependencies | RF-009 (campaignStore must be moved first; otherwise we'd wrap a moving target) |
| Files | 11 files → 1 persistence service (services/persistence/index.ts + 7 sub-files) |
| Estimated Wave | W7 |
