# 3.6 Traceability Matrix

**Input:** 3.1 RF catalog + 3.2 God File decomposition + 3.3 wave assignment
**Protocol:** Evidence-First — every RF case traces from Evidence → Design → Wave → Commit

## Matrix Purpose

This matrix is the **single source of truth** for refactor auditability.
For every RF case, it answers:

- **Why does this RF case exist?** → Evidence column (Phase 0 artifact)
- **What is the architectural answer?** → Design column (Phase 2 artifact)
- **When is it executed?** → Wave column (Phase 3.3)
- **Is it done?** → Status column
- **Where is the proof?** → Commit column (filled post-execution)
- **What was the result?** → Result column (filled post-execution)

This matrix is updated **after every wave** during Phase 4. A wave is
not considered closed until its RF cases show Status: Done and Commit
hash in this matrix.

## Status Vocabulary

Per Phase 3.5 review feedback, the matrix uses a 4-state vocabulary
to distinguish "infrastructure created" from "violation actually removed":

| Status | Meaning | When used |
|--------|---------|-----------|
| **Planned** | RF case is designed but no work done yet | Default state in 3.1 |
| **Prepared** | Infrastructure for the RF case exists (port/adapter/skeleton), but the violation is NOT yet removed | After W0 for RF-001..RF-007 |
| **In Progress** | Wave is actively executing the RF case closure | During the wave's branch life |
| **Done** | Violation is removed; smoke tests pass; gate.mjs confirms | After wave merge + matrix update |

**Critical distinction:** W0 set RF-001..RF-007 to **Prepared** (not Done),
because W0 created the ports but no service actually uses them yet.
The violations still exist. W1/W2/W3 will move them to Done.

## Post-Execution Update Protocol

After each wave's merge to `main`:

1. **Identify the merge commit hash** — recorded in the Wave Status Tracker
2. **Update each RF case row** that the wave closed:
   - Status: Pending → DONE
   - Commit: — → `<merge commit hash>`
   - Result: — → concrete outcome (e.g., "9 violations removed from campaignSlice; logic moved to services/campaign-state/")
3. **Update the Wave Status Tracker row**:
   - Status: Pending → DONE
   - Merge commit: — → `<hash>`
   - Violations after: verified via `node scripts/gate.mjs`
4. **Commit the matrix update** as a separate commit:
   `phase4: W{n} traceability matrix update — RF-XXX, RF-YYY DONE`

## Audit Query Examples

With this matrix populated, the following audits become trivial:

### "Why does commit abc123 exist?"
→ Look up abc123 in the Commit column. Find the RF case(s) it closes. Trace to Evidence column for the original problem, Design column for the architectural answer.

### "Which commits fixed the 9 campaignSlice violations?"
→ Look up RF-008. Wave = W4. Commit column has the merge commit. The merge commit's tree includes the campaignSlice extraction.

### "Is RF-009 actually complete?"
→ Look up RF-009. Status should be DONE. Commit should be non-empty. Result should describe concrete outcome. If any are missing, RF-009 is NOT complete — wave W5 was not properly closed.

### "What's the violation count after W5?"
→ Wave Status Tracker, row W5, column "Violations after" = 12.

### "Did any wave trigger self-correction?"
→ Self-Correction Audit Trail table. If empty, no self-correction occurred. If non-empty, each entry explains what was discovered and how the plan was revised.
