# 3.3 Wave Assignment & Dependency Graph (RF-based)

**Input:** 3.1 RF cases (RF-001..RF-019) + 3.2 God File splits
**Protocol:** Evidence-First — every wave is defined by which RF cases it closes, not by which tasks it performs

## Wave Definition Rule (Golden Rule)

> **A wave is authorized to run ONLY if it declares exactly which RF cases it closes (or advances).**

A wave is NOT a list of tasks ("build port", "build adapter", "wire DI").
A wave IS a list of RF outcomes ("close RF-001", "advance RF-009 to
infrastructure-ready state").

This rule has three consequences:

1. **No wave without RF cases.** If a wave's description doesn't name
   RF cases, the wave is invalid — it's task-based, not outcome-based.
2. **No RF case without a wave.** If an RF case isn't in any wave,
   it will never be executed. The wave assignment must cover all 19.
3. **Self-correction is mandatory.** If mid-wave we discover an RF
   case doesn't need its planned solution (e.g., RF-004 turns out
   not to need a port because the code can be refactored differently),
   the wave is revised, not forced through. The RF case is either
   re-scoped or moved to a different wave — never silently dropped.

## Wave Format

Each wave below has the following structure:

```
Wave W{n} — <name>
  Goal:           which RF cases this wave closes or advances
  Evidence:       Phase 0/1/2 artifacts that justify this wave
  Expected Result: specific violations removed, files changed,
                   observable coupling changes
  Files:          concrete file list
  Validation:     how "done" is verified
  Rollback:       how to undo
  Self-correction: what triggers a wave revision
```
