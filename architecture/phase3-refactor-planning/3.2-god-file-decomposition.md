# 3.2 God File Decomposition Plan

**Input:** 3.1 RF cases (RF-012..RF-019) + 0.16 line counts + 2.1 validations
**Protocol:** Evidence-First — splits are designed from actual function boundaries, not theoretical concerns

## Decomposition Principle

A God File is decomposed by **responsibility boundary discovered in the code itself**, not by an external pattern. We grep the file's exports, identify natural clusters, and split along those clusters.

Each split must satisfy:
1. **No cross-cluster calls hidden** — if file A's function calls file B's function, that's an explicit import, not a hidden dependency
2. **Testable in isolation** — each new file can be imported and tested without the others
3. **Re-export hub for backward compat** — original file becomes a barrel export, so existing imports don't break

## Split Execution Rules (all God Files)

1. **Barrel re-export first** — before any function moves, original file is converted to re-export. Build must still pass.
2. **One extraction per commit** — move one cluster, build, test, commit. Never batch.
3. **Smoke test before split** — for each God File, a smoke test exercising its primary flow must exist before W8/W9/W11 starts.
4. **No public API change** — all `export` statements from the original file must remain accessible from the original path (via barrel).
5. **Type signatures preserved** — extracted functions keep identical signatures.
