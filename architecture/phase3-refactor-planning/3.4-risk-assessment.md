# 3.4 Risk Assessment & Mitigation

**Input:** 3.1 RF cases + 3.2 God File splits + 3.3 wave assignment
**Protocol:** Evidence-First — risks derived from concrete code patterns, not generic fears

## Risk Model

Each risk is scored on two axes:
- **Likelihood** (1-5): how probable is this risk materializing
- **Impact** (1-5): how severe is the consequence if it does

Risk score = Likelihood × Impact (max 25). Mitigations are required
for any risk scoring ≥10.

## Risk Register

### R-01: Adapter delegation calls wrong store method

| Field | Value |
|-------|-------|
| ID | R-01 |
| Likelihood | 3 (medium — easy typo) |
| Impact | 5 (critical — silent behavior change) |
| Score | 15 |
| Affected RFs | RF-001..RF-007 |
| Affected waves | W0, W1, W2, W3 |
| Trigger | Adapter implements `appendMessage` but accidentally delegates to `setMessageImage` |
| Mitigation | (a) Adapter is auto-generated from port interface where possible (b) Smoke test each port method immediately after adapter is written (c) Code review checklist: every adapter method delegates to the exact method it's named after |
| Detection | Smoke tests in W0 must exercise every port method at least once |
| Recovery | `git revert <commit>` |

### R-03: God File split introduces circular imports

| Field | Value |
|-------|-------|
| ID | R-03 |
| Likelihood | 4 (high — common in TypeScript) |
| Impact | 3 (medium — build fails, not runtime) |
| Score | 12 |
| Affected RFs | RF-012, RF-013, RF-014, RF-017 |
| Affected waves | W8, W9, W10, W11e |
| Trigger | `npcGenerator.ts` calls `populateAgencyFields` (now in `npcAgency.ts`), but `npcAgency.ts` imports `npcGenerator.ts` for type definitions |
| Mitigation | (a) Extract shared types to `_shared.ts` (b) Extract shared utilities to `_shared.ts` first (c) Use barrel imports carefully — barrel itself can become a cycle (d) Run `madge --circular` after each extraction |
| Detection | `npx tsc -b` fails with circular import error |
| Recovery | Move the shared code to a third file; both importers depend on it |

|-------|-------|
| ID | R-04 |
| Likelihood | 2 (low — we control key names) |
| Impact | 5 (critical — users lose all data) |
| Score | 10 |
| Affected RFs | RF-009, RF-011 |
| Affected waves | W5, W7 |

## Open Questions

1. Should we add end-to-end Playwright tests?
   → Not for Phase 4 — they're expensive to maintain. Smoke tests +
     vitest unit tests are sufficient. Re-evaluate if W11e reveals
     integration issues.

2. Should we add a "fuzz" test for persistence?
   → Yes, in W7. Generate 100 random campaigns, save, reload, compare.

3. What's the rollback threshold?
   → If any wave fails 3 times in fix-forward mode, revert and
     re-evaluate the wave's design.
