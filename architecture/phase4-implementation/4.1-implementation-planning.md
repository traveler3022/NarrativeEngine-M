## Planning Principle

The implementation must be **reversible at every step**. We never make a
change that cannot be rolled back to the previous green build. Each wave
is a single commit (or a small set of commits), each commit passes the
full validation gate, and each wave is independently revertible.

The order is dictated by **RF case dependencies** (3.3), not by convenience:

1. **W0 — Infrastructure:** Ports and adapters come first because every
   later wave depends on them being available. If a service can't import
   a port, we can't extract logic from the store.
2. **W1-W3 — Service/slice migration:** Services and slices switch from
   direct imports to ports. Adapters delegate to the same store methods
   — no behavior change.
3. **W4-W6 — Logic extraction:** Store slices contain business logic
   (state→domain violations). Logic moves to services; slices become
   pure reactive state. Internal call patterns change; user-visible
   behavior preserved.
4. **W7 — Persistence consolidation:** 11 idb-keyval access points
   collapse into 1 persistence service.
5. **W8-W10 — God File splits (domain + state):** Large files split
   along responsibility boundaries discovered in the code. Barrel
   re-exports preserve public API.
6. **W11a-e — God Component splits:** Large components split into
   hooks + sub-components. W11e (ChatArea, TurnCallbacks provider
   change) runs last as the highest-risk component wave.
7. **W12 — Final validation:** Architecture diff confirms 53 → 0
   violations, 13 God Files → ≤4, 11 persistence access points → 1.

## Rollback Strategy

### Per-commit rollback
- Every commit must pass `tsc -b && vite build && vitest run` before push
- Commit message format: `phase4: W{n} <short description>` (updated from phase3)
- Single-purpose commits only — no "and also fixed X" mixes
- If a commit breaks the build, `git revert <sha>` is the canonical rollback

### Per-wave rollback
- Each wave lives on a branch `phase4/w{n}-<name>` branched from `main`
- Wave is merged to `main` only after passing all validation gates
- If a wave proves unsound, revert the merge commit on `main`

### Whole-phase rollback
- Tag `pre-phase3` (already created) preserves the entire pre-refactor state
- If Phase 4 fails catastrophically, `git reset --hard pre-phase3` restores
- This is the nuclear option — should never be needed if per-wave gates hold

### Behavior preservation contract
- **No behavior change** in W0 (pure infrastructure)
- **No behavior change** in W1-W3 (adapters delegate to same store methods)
- W4-W6 may change internal call patterns but must preserve user-visible
  behavior — verified by smoke tests
- W7-W11 may reorganize files but must not change observable behavior
  (barrel re-exports preserve public API)
- W11e changes the TurnCallbacks provider internally — the callback
  interface and observable behavior stay identical
- W12 is the verification that the contract held

## Validation Gates

### Per-commit gate (mandatory, blocking)
```bash
npx tsc -b --noEmit
npx vite build
npx vitest run
```
All three must pass. Any failure blocks the commit.

### Per-wave gate (mandatory, blocking)
```bash
# Per-commit gate (above)
node scripts/gate.mjs                    # no new violations vs previous wave
node scripts/baseline.mjs                # snapshot current architecture
npx madge --circular src/                # no new circular imports (for W8-W11)
node scripts/audit-exports.mjs <file>    # barrel re-exports match (for W8-W10)
```

## Wave Sizing Rules

- **Small waves preferred** — if a wave touches >10 files, split it
- **One responsibility per wave** — don't mix port migration with logic extraction
- **No wave depends on a later wave** — if it does, the dependency order is wrong
- **Each wave's success is independently verifiable** — if you can't describe
  what "done" looks like for a wave, the wave is too vague
